# How the M262 `.hcf` got into EAE's Hardware view

## Symptom

`Devices > M262 > M262_RES` in EAE was blank after Button 2. No TM3 modules,
no symbolic links â€” even though the syslay/sysres looked fine.

## Root cause (three independent gaps)

1. **The deployed `.hcf` was the empty shell `<DeviceHwConfigurationItems />`.**
   `DemonstratorWiper` writes that empty shell on Clean Demonstrator, and
   `M262HwConfigCopier.Copy` (which should reseed it from baseline inside
   `FinalizeM262StackAsync`) was failing silently inside a try/catch â€” so
   the patch service had no `<TM3DI16_G>` / `<TM3DQ16T_G>` modules to
   write `ParameterValue`s into.

2. **The deployed `.sysres` had an empty `<FBNetwork />` â€” no `M262IO` FB
   instance.** Even if the `.hcf` had references like `RES.FB.PusherAtHome`,
   EAE couldn't resolve the middle term to anything, so the Symbolic Links
   table stayed blank. The variable mapping is:
   `{resource-id}.{m262io-fb-id}.{varName}` â†’ EAE looks up the FB by ID
   inside the sysres FBNetwork.

3. **Wrong file name convention.** I'd been writing the `.hcf` at
   `{sysdev-dir}/{resource-guid}.hcf`, but EAE looks for
   `{sysdev-dir}/{sysdev-guid}.hcf` â€” file stem matches the *folder name*
   (sysdev guid), not the resource guid. The resource guid only appears as
   the `ResourceId` attribute *inside* the XML.

## Fix (`MapperUI/MapperUI/Services/HcfPatchService.cs`)

Replaced the "load existing .hcf, patch ParameterValues" approach with a
direct write of an Alex-shaped `.hcf` template every Button 2 run:

1. **Locate the M262 sysdev** by walking
   `{Demonstrator}/IEC61499/System/` for a `.sysdev` whose root `<Device>`
   has `Type="M262_dPAC"` and `Namespace="SE.DPAC"`. Read the embedded
   `<Resources><Resource Name="M262_RES" ID="..."/>` to get the **resource
   guid**.

2. **Inject the `M262IO` FB into the sysres** if missing
   (`EnsureM262IoFb`). Generates a deterministic 16-char hex ID via SHA-1
   of `"M262IO|" + resourceId` so the same input gives the same FB ID
   across runs (`NewShortHexId`). EAE accepts both long-dashed GUIDs and
   short hex; baselines use the short form.

3. **Compose the target path** as
   `{sysdev-dir}/{sysdev-guid}.hcf` â€” file stem = sysdev folder name.
   Delete any stale `.hcf` in the same folder with a different stem.

4. **Build the `.hcf` XML verbatim** (`BuildHcfXml`) with:
   - `BMTM3` master (busid=TM3Config, busCycleTime=T#80ms, enableSymlinkOC=TRUE)
   - `TM262L01MDESE8T` CPU
   - `TM3DI16_G` with all 16 channels (`Channel_0..15.Latch=32`, `.Filter=4`)
     and `ParameterValue` entries `DI00..DI15`
   - `TM3DQ16T_G` with `ParameterValue` entries `DO00..DO15`
   - Substitute `{resource-id}.{m262io-fb-id}.{tag}` into every bound pin:
     - `DI00..DI09` â†’ PusherAtHome, PusherAtWork, Hopper, CheckerUp,
       CheckerDown, PartAtChecker, TransferAtHome, TransferAtWork,
       PartAtAssembly, PartAtExit
     - `DO00..DO03` â†’ ExtendPusher, ExtendChecker, ExtendTransfer, ExtendRejector
     - All other pins blank

5. **Activity log lines** before and after the write print the absolute
   target path so you can grep the panel without parsing the file system.

## Why EAE then renders it as `'M262_RES.M262IO.PusherAtHome'`

The on-disk `.hcf` stores GUIDs (`{resource-guid}.{fb-guid}.PusherAtHome`).
EAE resolves the resource guid â†’ resource name (`M262_RES`) and the FB guid
â†’ FB name (`M262IO`) for display. The single quotes are EAE's UI treatment
of a symbolic-link literal. So the file on disk uses GUIDs but the
Symbolic Links view shows friendly names â€” that's by design.
